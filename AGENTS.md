# AGENTS.md

本文件定义 Codex 在本仓库中的工作约定。作用范围为当前项目根目录及其所有子目录。

## 1. 目标与原则

- 目标：高质量、可验证地完成运筹优化相关代码与文档工作。
- 原则：先理解后修改、最小必要改动、结果可复现、过程可追踪。
- 默认语言：中文沟通，代码与变量名必须使用英文。

## 2. 标准工作流（必须遵守）

1. 理解任务  
   - 先确认需求边界、影响范围、交付物。  
   - 如存在歧义，优先基于仓库上下文做合理假设并继续推进。

2. 建立上下文  
   - 先阅读相关目录、脚本与 README。  
   - 搜索文件与文本优先使用 `rg` / `rg --files`。

3. 设计改动  
   - 优先复用现有抽象（当前仓库以 `src/ModelSolver.py` 为统一模型接口）。  
   - 新增能力时保持向后兼容，避免无关重构。

4. 实施改动  
   - 小步提交式修改，单次改动聚焦一个问题。  
   - 非必要不改命名、不改目录结构、不改公共接口。

5. 自检与验证  
   - 至少执行语法检查（如 `py_compile`）。  
   - 对关键脚本执行运行验证，确认输出状态/目标值/变量值合理。  
   - 若无法运行测试，需明确说明原因与风险。

6. 结果交付  
   - 给出变更摘要、影响文件、验证结果、后续建议。  
   - 用户未明确要求时，不执行 `push`。

## 3. 编码与文档规范

- Python 代码遵循：
  - 明确函数职责，避免过长函数。
  - 优先类型标注与可读命名。
  - 仅在必要处添加简洁注释，避免噪音注释。
- 文档遵循：
  - Markdown 格式文档使用 UTF-8 编码。
  - 数学模型优先使用公式表达（目标函数、约束、变量定义），Markdown 格式文档中使用 Latex 数学公式表达。
  - 保持目录说明与实际文件一致。

## 4. 优化模型约定（本仓库特定）

- 统一通过 `OptimizationModel` 描述模型。
- `problem_type` 必须按模型类型标注（如 `LP`、`IP`、`MIP`、`QP`、`MIQP`、`NLP`、`MOIP`）。
- 变量真实边界使用 `lb/ub`；仅供参考枚举的搜索范围使用 `search_lb/search_ub`，不得混作数学约束。
- 旧脚本兼容入口可以保留，但应作为薄封装，核心逻辑放在统一模型脚本中。

## 5. Git 与发布规则

- 不回滚用户已有改动，除非用户明确要求。
- 不使用破坏性命令（如 `git reset --hard`）。
- 提交信息需清晰描述改动目的与范围。
- 仅当用户明确说 `push code` 时执行推送。

## 6. 沟通规则

- 先同步“将做什么”，再执行。
- 任务进行中保持简短进度更新。
- 结束时明确：
  - 改了什么
  - 验证了什么
  - 还有什么风险或下一步建议

## 7. 任务模板（建议复用）

### 7.1 需求输入模板

```text
【任务目标】
例如：重构 src/xxx 目录中的模型脚本，统一为 ModelSolver 接口。

【范围】
例如：仅修改 src/xxx 与对应 README，不改其他目录。

【约束】
例如：保留旧脚本文件名作为兼容入口；不删除历史示例。

【验收标准】
例如：
1. 脚本可运行并输出结果
2. README 包含数学模型公式
3. py_compile 检查通过

【发布要求】
例如：先提交，不推送；我确认后再 push code。
```

### 7.2 验收清单（交付前自检）

- `范围`：只改了需求范围内文件，无无关改动。
- `正确性`：模型目标函数、约束、变量类型与原问题一致。
- `一致性`：`problem_type`、目录分类、README 说明相互一致。
- `可运行性`：关键脚本已执行，状态与结果合理。
- `可维护性`：公共逻辑进入公共模块，入口脚本保持薄封装。
- `编码`：文件为 UTF-8，中文文档与注释无乱码。
- `交付说明`：已给出变更摘要、验证结果、未完成项/风险。

## 8. 对齐 Codex 官方最佳实践

基于 OpenAI Codex 官方 best practices（https://developers.openai.com/codex/learn/best-practices），本仓库增加以下约束：

- 指令要具体：任务必须写清目标、范围、约束、验收标准；优先使用本文件第 7 节模板。
- 上下文要充分：先读取相关代码与文档，再修改；避免脱离仓库上下文给出方案。
- 计划与执行分离：先说明执行路径，再进行改动；复杂任务要分步骤推进并持续同步进度。
- 优先可验证输出：每次改动后给出可复现验证（命令、状态、关键结果）。
- 小步安全变更：避免一次性大改，优先可回滚的小步提交。
- 明确工具边界：涉及高风险命令、外部依赖、不可逆操作时先确认。
- 文档与代码同步：接口变化必须同步 README/注释，避免知识漂移。

## 9. Codex 工作流目录

按 Codex 官方最佳实践，首选使用 `.codex/` 目录存放可复用工作流资产：

- `.codex/TASK_TEMPLATE.md`：任务输入模板
- `.codex/REVIEW_CHECKLIST.md`：交付前检查清单
- `.codex/RUNBOOK.md`：仓库常用执行与验证命令
